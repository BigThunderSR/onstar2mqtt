# Contributing to OnStar2MQTT

## Pull Requests

### Diagnostic Data Requirements

All diagnostic-related PRs **must** use real data from the OnStar API. Specifically:

- **Test fixtures** must be captured from an actual vehicle's diagnostic response — not fabricated, not generated by AI, not copied from documentation.
- **Element names** must match the exact strings returned by the API (e.g., `BRAKE_PAD_LIFE_FRONT`, not `BRAKE_PAD_LIFE` or `EE_BPLM`).
- **Data types** must be verified against real responses. Numeric values get `state_class: measurement`; string values (like `"OK"` or `"TRUE"`) do not.
- **Sensitive data** (VIN, location, tokens) should be redacted from fixtures, but the structure and field names must remain authentic.

If you don't have access to a vehicle with the relevant diagnostic group, open an issue requesting it and someone with access can provide sanitized data.

### Code Quality

- All tests must pass: `npm test`
- Lint must pass: `npm run lint`
- New sensor types need test coverage with specific assertions on the generated HA config (icon, device_class, state_class, unit_of_measurement)

### AI-Generated Code

AI tools can help write boilerplate, but the final PR must be verified against reality. PRs containing fabricated test data or hallucinated API element names will be closed without merge.
